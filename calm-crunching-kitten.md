# DOE 분석 하네스 — 핸드오프 노트

> 이 문서는 Claude Code에서의 검토 결과를 코워크 환경으로 옮기기 위한 핸드오프다. 다른 세션·다른 도구에서 cold start로 읽어도 이어갈 수 있도록 자기완결적으로 작성했다.

---

## TL;DR

코딩 못 하는 동료들을 위한 **데이터 분석 스킬 하네스**를 만들려 했다. 가장 큰 갈등은 *"ChatGPT/Claude.ai가 이미 파일 업로드·분석·차트를 잘 하는데 이걸 만들 이유가 있나"*였다. 이 갈등을 정직하게 풀어본 결과, **데이터 분석 일반으론 패배. 단 DOE(실험 결과 해석)로 도메인을 좁히면 차별화가 명확함**이라는 결론에 이르렀다. 만들기 전 ChatGPT 대비 bake-off로 thesis를 검증하고, factorial DOE 단일 케이스로 PoC를 시작한다.

## 1. 배경

- **만드는 사람(나)**: 코딩 가능. 본인용은 아님.
- **타겟 사용자**: 코딩을 못 하는 동료들. csv/엑셀을 올려 데이터 분석가의 관점에서 도움받는 형태.
- **환경**: Claude Code를 코딩 도구가 아니라 *데이터 작업장*으로 사용. 스킬 하네스가 분석 SOP를 코드화해 동료들에게 제공.
- **이전 상태**: DESIGN.md(687줄), debate 문서(1750줄)를 작성했으나 범위가 너무 넓어 꼬임. 이 핸드오프는 그 문서를 무시하고 처음부터 다시 정리한 결과다.

## 2. 고민했던 부분 — ChatGPT/Claude.ai와의 차별화 갈등

핵심 질문은 단 하나였다.

> **"ChatGPT와 Claude.ai가 이미 파일 업로드·분석·차트 생성을 잘 한다. 같은 일을 하네스로 다시 만들 이유가 있는가? 동료들이 그냥 ChatGPT 쓰면 되는 거 아닌가?"**

이걸 정직하게 검토한 결과:

### 2-1. Web 도구(ChatGPT/Claude.ai)가 더 잘하는 영역
- 일회성 탐색 분석 (EDA, ad-hoc 질문)
- 일반적 통계 테스트, 빠른 차트 그리기
- 업로드 한도 안의 파일에 대한 setup-free 분석
- 모델 자체의 분석 능력 — 같은 Claude/GPT라 **천장이 같음**

이 영역에서 하네스는 거의 못 이긴다. setup 비용만 더 크다.

### 2-2. 하네스 + Claude Code가 본질적으로 우위인 영역
- 반복 표준 분석의 **SOP화** (조직 고유 방식의 코드화)
- 결과물의 **재현성/자산화** (채팅 로그가 아닌 로컬 파일·노트북·git)
- **도메인 특화** (좁은 페르소나에 맞춘 어휘·지표·표준 테스트)
- **민감 데이터 로컬 보존** (외부 업로드 불가 데이터에선 대체 불가)
- **분석→리포팅→배포 파이프라인** (MCP로 Notion/Slack 발행)

### 2-3. 결정적 자가검증 질문
> "이걸 그냥 ChatGPT에 던지면 안 되는 이유는?"

분석 케이스마다 30초 내에 답이 안 나오면 그 케이스는 하네스 영역이 아니다. 매번 이 질문으로 거름.

### 2-4. 결론
**"데이터 분석 일반"으로 가면 진다.** 차별화 축 1~2개로 좁혀야 한다. 이전 시도가 꼬인 이유도 동일 — 범위 미좁힘.

## 3. 도메인을 DOE로 좁힌 이유

가능한 차별화 축들 중 **DOE(Design of Experiments) / 실험 결과 해석**을 선택했다.

### 3-1. DOE가 차별화 가능한 이유 (이게 핵심)

DOE는 ChatGPT가 **가장 흔들리는 분석 도메인 중 하나**다. 다음 3가지가 동시에 발생한다:

1. **필수 진단 누락 (silent error)**
   - DOE에는 reside normality, lack of fit, homoscedasticity, residual independence 같은 필수 진단이 있다.
   - ChatGPT는 비전문가 프롬프트엔 이걸 자주 빠뜨린다.
   - 비전문 동료가 이걸 catch 못 함 → 잘못된 결론을 신뢰함.

2. **해석이 일반론으로 흐름**
   - "유의미해 보입니다" 수준의 답.
   - DOE는 *main effect / interaction effect / 효과 크기(η², Cohen's f) / 신뢰구간*이 구조화돼 보고돼야 함.
   - LLM은 명시적 SOP 없이는 일반론으로 답한다.

3. **Metadata 휘발**
   - factor 이름·단위·수준 같은 실험 metadata가 매 세션 사라짐.
   - 하네스는 프로젝트 파일에 영속.

여기에 **R&D/제조 데이터는 외부 업로드 금지** 케이스가 많아 로컬 처리 자체가 본질적 우위.

### 3-2. 표면은 "일회성", 절차는 "표준화"
사용자 본인은 이 분석을 "일회성 EDA"로 분류했지만, 실제 DOE는 절차가 매우 표준화된 분야다(factorial, fractional factorial, RSM, mixture, Taguchi 등). **표면적 일회성 ≠ SOP 불가**. 오히려 SOP 코드화 가치가 매우 큼.

## 4. 차별화 Thesis (한 문장)

> **"DOE 비전문가 동료가 ChatGPT에 실험 데이터를 던졌을 때 발생하는 silent error(필수 진단 누락, 일반론적 해석, metadata 휘발)를 강제 SOP와 도메인 어휘로 차단한다."**

이 thesis가 살아있는지를 모든 결정에서 자가검증한다.

| 차별화 축 | ChatGPT의 약점 | DOE 하네스의 우위 |
|---|---|---|
| 진단 완전성 | 필수 진단 4종 자주 누락 | 체크리스트 강제 실행 |
| 해석 구조 | 일반론적 결론 | main/interaction effect, 효과 크기, CI 표준 출력 |
| Metadata 일관성 | 매 세션 휘발 | 프로젝트 파일에 영속 |
| 재현성 | 채팅 로그 = 비자산 | 노트북·보고서·git 커밋 = 자산 |
| 데이터 보안 | 외부 업로드 필요 | 로컬에서 종결 |

## 5. 검증 방법 — Bake-off (만들기 전에)

코드 한 줄 쓰기 전에 thesis를 측정한다. 이게 안 되면 또 꼬인다.

**절차:**
1. 공개 DOE 데이터셋 3~5개 준비 (Box & Draper, Montgomery 교재 예제 — 정답이 알려진 것)
2. 동료 페르소나로 ChatGPT/Claude.ai에 단순 프롬프트 사용 ("이 실험 결과 분석해줘")
3. 다음 항목 채점 (각 0~10점):
   - 필수 진단 4종 모두 수행했나?
   - main effect와 interaction effect를 구분해 보고했나?
   - 효과 크기를 양적으로 보고했나(η², Cohen's f)?
   - 결론이 구체적인가, 일반론인가?
   - 동료 1~2명이 어느 쪽을 신뢰하는가?
4. **Go/No-Go**:
   - ChatGPT 평균 7/10↑ → thesis 약함. 좁힘 더 필요하거나 포기.
   - ChatGPT 평균 4/10↓ → thesis 강함. PoC 진행.

bake-off 결과는 향후 모든 설계 결정의 reference.

## 6. PoC 범위 (의도적으로 좁게)

### 1차 PoC: **Factorial DOE 결과 분석 단 하나만**

워크플로우:
1. csv 업로드 (factor 컬럼 + response 컬럼)
2. design 구조 자동 추정 + 동료에게 컨펌 (2^k full? fractional? center points?)
3. 표준 분석: ANOVA → main/interaction effect → 효과 크기 → Pareto chart → main effect plot → interaction plot
4. 진단 자동 실행: Shapiro / residual vs fitted / Levene / lack of fit
5. 표준 보고서: Markdown + 그래프 PNG, 결론·진단 결과·다음 실험 권장 포함

### 1차에서 명시적 제외
- 일반 EDA (ChatGPT가 빠름 — 이기려 들지 말 것)
- 일반 통계 테스트, 일반 regression
- Power analysis, 사전 design 생성
- RSM, mixture, Taguchi (PoC 통과 후 확장 후보)

## 7. 단계별 Go/No-Go

| Step | 내용 | 통과 기준 |
|---|---|---|
| 1 | 동료 1~2명에게 "최근 6개월 DOE 빈도" 인터뷰 | 월 1회 이상 |
| 2 | Bake-off 채점 | ChatGPT 4/10↓ |
| 3 | factorial PoC 단일 스킬 구축 | reference 결과와 수치 일치 |
| 4 | 동료 사용자 테스트 5건 | "ChatGPT 대신 쓸 의향" 70%↑ |
| 5 | 확장(RSM 등) 또는 thesis 재검토 | — |

**각 Step에 통과 기준을 박는 것이 핵심.** 이전 시도가 꼬인 가장 큰 이유는 통과 기준 부재.

## 8. 리스크

- **동료가 실제론 DOE를 거의 안 함**: 가장 큰 리스크. Step 1에서 빈도 확인 필수.
- **Python(statsmodels) 정확도가 JMP/Minitab 미달**: 산업 표준과 수치 비교로 검증해야 함.
- **최신 ChatGPT가 의외로 잘함**: bake-off가 약하게 나오면 정직하게 멈출 것.
- **DOE 도메인 자체가 협소함**: 확장 로드맵(RSM, mixture, Taguchi)이 동료의 실제 워크플로우와 맞는지 Step 4에서 같이 물어볼 것.

## 9. 코워크에서 다듬을 항목 (제안)

코워크로 옮겨가서 다음을 구체화하면 좋겠다:

1. **동료 페르소나 정밀화**
   - 직무, 부서, DOE 빈도, 현재 사용 도구(JMP / Minitab / Excel / 직접 R)
   - "현재 ChatGPT를 쓰는가? 안 쓴다면 왜?" — 핵심 질문
2. **Bake-off 채점표 양식**
   - 각 항목의 가중치, 평가자 다중화 여부
3. **PoC 보고서 템플릿 초안**
   - 섹션 구성, 차트 종류, 결론 톤(보수적 / 단정적), 다음 실험 권장 형식
4. **사용자 테스트 설계**
   - "ChatGPT 대신 쓸 의향"을 묻는 정확한 문구, 비교 시나리오
5. **확장 로드맵 우선순위**
   - RSM vs mixture vs Taguchi vs Plackett-Burman — 동료 실제 사용 빈도 기준
6. **Privacy/배포 모드**
   - 로컬 단독? 팀 공유 git repo? 사내 마켓플레이스?
